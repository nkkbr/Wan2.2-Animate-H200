# Step 04: H200 运行路径与静态条件缓存

## 1. 目标

针对单张 H200 141GB GPU，建立一条“质量优先”的 generate 运行路径，并把当前明显重复计算的静态条件改成缓存。

这一步的核心目标有两个：

- 把 H200 的资源真正转化为质量预算
- 为后续大量实验降低时间成本


## 2. 为什么现在做

当前单卡默认逻辑对 animate 任务并不理想：

- `offload_model` 默认会被自动设为 `True`
- 每个 clip 都会重复编码参考图和文本相关条件

这会带来两个问题：

1. 推理时间被不必要的 CPU/GPU 搬运和重复编码浪费。
2. 后续进行大规模参数搜索和画质实验时，实验成本被显著抬高。

对于 H200 来说，这种默认策略过于保守。


## 3. 本步骤的范围

本步骤包括：

- H200 高质量运行 preset
- 关闭无谓 offload
- 缓存静态文本与参考图条件
- 增加显存、时间和 clip 级性能日志

本步骤不做：

- continuity 逻辑修改
- seam blending
- soft mask
- background inpainting


## 4. 当前重复计算点

在当前 animate replacement 逻辑中，整段视频中不变但会被重复算的东西包括：

- `context`
- `context_null`
- `ref_latents`
- `mask_ref`
- `y_ref`
- `clip_context`

这些量与 clip index 无关，却会在每个 clip 中重复参与构建或重复编码。

对长视频来说，这是很不经济的。


## 5. 设计原则

### 5.1 不改变数值语义

缓存应尽量做到：

- 与当前逻辑数值等价
- 不改变模型输入的语义

这样可以把风险控制在最小范围。

### 5.2 H200 路径应显式可选

不要把所有用户都强行切到 H200 高质量路径。  
更合理的方式是：

- 增加高质量 preset
- 或增加清晰的 CLI 参数

### 5.3 让性能日志成为后续决策依据

这一步不仅要“更快”，还要能看见：

- 每个 clip 的耗时
- 参考条件编码耗时
- VAE encode/decode 耗时
- 峰值显存


## 6. 具体实施方案

### 6.1 增加 H200 高质量 preset

建议增加统一入口，例如：

- `--quality_preset hq_h200`

该 preset 至少包含：

- `offload_model=false`
- 推荐的 solver / steps 默认值
- 开启更详细的 debug 日志

如果暂时不想做 preset，也至少要把 H200 场景下的推荐命令写进主文档和示例脚本。

### 6.2 缓存文本条件

对整段视频来说，下列内容只需编码一次：

- 正文本 context
- 负文本 context_null

建议：

- 在 `generate()` 开始时一次性编码
- 后续 clip 循环只复用

### 6.3 缓存参考图条件

参考图相关的静态条件包括：

- `ref_latents`
- `mask_ref`
- `y_ref`
- `clip_context`

建议：

- 在 clip 循环外一次性计算
- 后续所有 clip 直接复用

### 6.4 增加性能日志

建议至少记录：

- clip index
- clip 耗时
- scheduler 步数
- VAE encode/decode 耗时
- 静态条件编码耗时
- 峰值显存

必要时把这些信息同时写入 run manifest。


## 7. 需要改动的文件

建议涉及：

- `generate.py`
- `wan/animate.py`

必要时辅助：

- 运行日志工具
- manifest 写入逻辑


## 8. 如何实施

建议顺序如下：

1. 先加高质量 preset 或等价参数方案。
2. 再把文本条件提到 clip 循环外。
3. 再把参考图条件提到 clip 循环外。
4. 最后增加性能日志和显存统计。

注意不要一开始就把更多动态条件一起缓存。  
当前这一步只缓存“真正静态”的条件，避免引入语义错误。


## 9. 如何检验

### 9.1 数值等价检验

在相同 seed、相同参数下，对改动前后结果做对比：

- 如果缓存逻辑正确，输出应与原逻辑一致或仅有极小数值误差

### 9.2 性能检验

至少比较：

- 总生成时间
- 单个 clip 平均时间
- GPU 峰值显存

### 9.3 稳定性检验

在长视频 benchmark 上确认：

- 关闭 offload 后没有显存溢出
- 缓存逻辑不会导致跨 clip 状态污染


## 10. 验收标准

只有满足下面条件，才算本步骤完成：

- H200 路径可以显式开启
- 文本和参考图静态条件不再重复编码
- 结果与旧逻辑保持一致
- 性能日志完整可用
- 在 benchmark 上稳定运行


## 11. 风险与回退

### 风险

- 误把动态条件缓存成静态条件
- 关闭 offload 后某些极端分辨率组合显存压力上升

### 回退策略

- 缓存逻辑逐项启用，每项单独验证
- 保留 `offload_model=true` 的手动 fallback


## 12. 本步骤完成后的收益

这一步完成后，后续几乎所有质量实验都会更便宜：

- 更高 `sample_steps` 的成本更可接受
- 大量参数搜索会更快
- 更容易把预算投入到真正影响质量的步骤，而不是重复编码静态条件
